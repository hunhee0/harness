# 프론트엔드 디자인 방향

UI를 단순히 동작시키는 것이 아니라 의도가 있고 폴리싱되고 제품 도메인에 적절하다고 느끼게 만드는 작업에 이 스킬 사용.

소스: `linus707`의 stale 커뮤니티 PR #1659에서 구조.

비고: ECC는 의도적으로 정식 Anthropic `frontend-design` 스킬을 재번들하지 않음. 공식 업스트림 스킬을 원하면 `anthropics/skills`에서 설치. 이 스킬은 #1659의 유용한 로컬 가이던스의 ECC 특화 디자인-방향 구조본임.

## 사용 시점

- 사용자가 웹 페이지·앱·대시보드·아티팩트·컴포넌트·UI 구축 요청
- 사용자가 인터페이스를 더 폴리싱·독창·아름답게·덜 generic하게 요청
- 구현에 시각 계층·타이포그래피·색상·모션·레이아웃·상호작용 선택 필요
- 현재 UI가 동작하지만 평면·generic·템플릿·청중 부적합으로 읽힘

## 디자인 방향

코딩 전 구체적 방향 선택:

1. 목적: 인터페이스가 어떤 일을 하는가?
2. 청중: 이 워크플로를 반복하는 사람은 누구이며, 무엇을 먼저 스캔해야 하는가?
3. 톤: 실용적·편집적·놀이적·산업적·세련된·기술적·맥시멀·미니멀·dense·calm 또는 다른 명시적 방향.
4. 기억할 디테일: 결과를 의도적으로 느끼게 만드는 디자인 아이디어 하나.
5. 제약: 프레임워크·접근성·성능·반응성·기존 디자인 시스템.

방향을 도메인에 맞춤. SaaS 운영 도구는 보통 dense·quiet·스캔 가능해야 함. 포트폴리오·런치 페이지·게임·편집 작품은 더 표현적일 수 있음. 매일 반복 사용이 필요한 도구에 랜딩 페이지 컴포지션을 강요하지 말 것.

## 구현 가이드

- 사용자가 명시적으로 마케팅 카피를 요청하지 않는 한 실제 사용 가능한 경험을 첫 화면으로 구축.
- 새 시각 시스템 도입 전에 기존 프로젝트 컴포넌트·토큰·아이콘 라이브러리·라우팅 패턴 사용.
- 인터페이스가 이미지·제품·장소·사람·게임플레이·차트·검사 가능 미디어에 의존하면 실제 또는 생성된 시각 자산 사용.
- generic한 거대 hero 텍스트보다 컨텍스트 타이포그래피·간격 선호.
- 팔레트를 다차원으로 유지: 단일 색상 계열이 지배하는 UI 회피.
- 방향이 상태 전반에 일관되도록 CSS 변수·기존 디자인 토큰 사용.
- 반응형 제약을 명시적으로 디자인: 그리드·종횡비·min/max 크기·안정 툴바·고정 형식 컨트롤이 라벨·hover 상태 출현 시 이동하지 않아야 함.
- 모션은 절제하되 의도적으로 사용. 장식 애니메이션보다 상태를 명확히 하는 고신호 전환 선호.
- 모바일·데스크톱에서 텍스트 fit 검증. 긴 라벨은 오버플로 대신 깔끔히 wrap 또는 resize되어야 함.

## 안티패턴

- 흔한 생성 패턴 기본값 회피: purple gradient·장식 blob·거대 카드·모호한 hero 카피·stock 같은 분위기 미디어.
- 카드 안에 UI 카드 추가 금지.
- 도메인이 절제를 요구할 때 모든 곳에 단일 장식 스타일 사용 금지.
- generic한 마케팅 섹션 뒤에 주된 제품·도구·객체·워크플로 숨기기 금지.
- 디자인 plourish용 새 의존성 추가는 명확히 본전을 뽑지 않으면 금지.
- 컨트롤이 스스로 말할 수 있는데 UI 안에서 UI 기능 설명 금지.

## 리뷰 체크리스트

- 첫 viewport가 즉시 제품·워크플로·객체를 전달.
- 시각 계층이 스캔·반복 사용을 지원.
- 타이포그래피가 컨테이너에 fit하고 인접 컨텐츠와 겹치지 않음.
- 색 선택에 대비 있음, 단조로운 팔레트로 붕괴 안 함.
- 친숙한 도구 액션에 아이콘 사용 가능 시 사용.
- 반응형 레이아웃이 보드·그리드·툴바·컨트롤·타일·카운터에 안정 차원 보유.
- 자산이 렌더링되고 채움이 아닌 주제를 담음.
- 모션이 orientation 개선, 느림 마스킹 안 함.
- 명확한 이탈 사유가 없는 한 결과가 레포의 기존 프론트엔드 관례와 일치.
