## 사용자 입력

```text
$ARGUMENTS
```

진행 전 반드시 사용자 입력을 고려하세요 (비어있지 않은 경우).

## 개요

`.specify/memory/constitution.md`의 프로젝트 헌법을 업데이트합니다. 이 파일은 대괄호 안에 플레이스홀더 토큰이 포함된 **템플릿**입니다 (예: `[PROJECT_NAME]`, `[PRINCIPLE_1_NAME]`). 작업은 (a) 구체적인 값을 수집/도출하고, (b) 템플릿을 정확하게 채우고, (c) 수정 사항을 종속 아티팩트에 전파하는 것입니다.

**참고**: `.specify/memory/constitution.md`가 아직 없는 경우, 프로젝트 설정 중에 `.specify/templates/constitution-template.md`에서 초기화되어 있어야 합니다. 없는 경우 먼저 템플릿을 복사하세요.

다음 실행 흐름을 따르세요:

1. `.specify/memory/constitution.md`의 기존 헌법을 로드합니다.
   - `[ALL_CAPS_IDENTIFIER]` 형식의 모든 플레이스홀더 토큰 식별.
   **중요**: 사용자는 템플릿에 사용된 것보다 적거나 많은 원칙을 요구할 수 있습니다. 숫자가 지정된 경우 그것을 존중하고 일반 템플릿을 따르세요. 문서를 그에 맞게 업데이트합니다.

2. 플레이스홀더 값 수집/도출:
   - 사용자 입력(대화)이 값을 제공하면 사용.
   - 그렇지 않으면 기존 저장소 컨텍스트(README, docs, 이전 헌법 버전이 포함된 경우)에서 추론.
   - 거버넌스 날짜의 경우: `RATIFICATION_DATE`는 원래 채택 날짜 (모르면 묻거나 TODO로 표시), `LAST_AMENDED_DATE`는 변경 시 오늘 날짜, 그렇지 않으면 이전 값 유지.
   - `CONSTITUTION_VERSION`은 시맨틱 버전 관리 규칙에 따라 증가해야 합니다:
     - MAJOR: 하위 호환성 없는 거버넌스/원칙 제거 또는 재정의.
     - MINOR: 새 원칙/섹션 추가 또는 실질적으로 확장된 가이드.
     - PATCH: 명확화, 표현, 오타 수정, 비의미적 개선.
   - 버전 범프 유형이 모호한 경우 확정 전에 근거 제안.

3. 업데이트된 헌법 내용 초안 작성:
   - 모든 플레이스홀더를 구체적인 텍스트로 교체 (프로젝트가 아직 정의하지 않기로 선택한 의도적으로 유지된 템플릿 슬롯 제외 — 명시적으로 정당화).
   - 헤딩 계층 구조와 주석은 교체 후 제거 가능하지만 여전히 명확화 가이드를 제공하는 경우 유지.
   - 각 원칙 섹션 보장: 간결한 이름 줄, 협상 불가 규칙을 담은 단락(또는 불릿 목록), 명확하지 않은 경우 명시적 근거.
   - 거버넌스 섹션에 수정 절차, 버전 관리 정책, 규정 준수 검토 기대사항 나열.

4. 일관성 전파 체크리스트 (이전 체크리스트를 능동적 검증으로 변환):
   - `.specify/templates/plan-template.md`를 읽고 "헌법 확인" 또는 규칙이 업데이트된 원칙과 정렬되어 있는지 확인.
   - `.specify/templates/spec-template.md`의 범위/요구사항 정렬 확인 — 헌법이 필수 섹션이나 제약을 추가/제거하는 경우 업데이트.
   - `.specify/templates/tasks-template.md`를 읽고 태스크 분류가 새로운 또는 제거된 원칙 주도 태스크 유형을 반영하는지 확인 (관찰 가능성, 버전 관리, 테스트 규율 등).
   - `.specify/templates/commands/*.md`의 각 명령 파일을 읽고 (이 파일 포함) 일반적인 가이드가 필요한 경우 오래된 에이전트별 이름(CLAUDE만 등)이 남아 있지 않은지 확인.
   - 런타임 가이드 문서(예: `README.md`, `docs/quickstart.md`, 또는 있는 경우 에이전트별 가이드 파일) 읽기. 변경된 원칙에 대한 참조 업데이트.

5. 동기화 영향 보고서 생성 (업데이트 후 헌법 파일 상단에 HTML 주석으로 첨부):
   - 버전 변경: 이전 → 새로
   - 수정된 원칙 목록 (이름 변경 시 이전 제목 → 새 제목)
   - 추가된 섹션
   - 제거된 섹션
   - 업데이트가 필요한 템플릿 (✅ 업데이트됨 / ⚠ 대기 중) 파일 경로 포함
   - 의도적으로 연기된 플레이스홀더가 있는 경우 후속 TODO.

6. 최종 출력 전 검증:
   - 설명되지 않은 괄호 토큰 없음.
   - 버전 줄이 보고서와 일치.
   - 날짜 ISO 형식 YYYY-MM-DD.
   - 원칙이 선언적이고 테스트 가능하며 모호한 언어 없음 ("should" → 적절한 경우 MUST/SHOULD 근거로 교체).

7. 완성된 헌법을 `.specify/memory/constitution.md`에 씁니다 (덮어쓰기).

8. 사용자에게 최종 요약 출력:
   - 새 버전 및 범프 근거.
   - 수동 후속 조치가 필요한 파일.
   - 제안된 커밋 메시지 (예: `docs: amend constitution to vX.Y.Z (principle additions + governance update)`).

형식 및 스타일 요구사항:

- 템플릿과 정확히 동일한 Markdown 헤딩 사용 (레벨 올리거나 내리지 않음).
- 가독성을 위해 긴 근거 줄 줄바꿈 (<100자 이상적) 하지만 어색한 중단으로 강제 적용 안 함.
- 섹션 사이에 빈 줄 하나.
- 후행 공백 피하기.

사용자가 부분 업데이트를 제공하는 경우 (예: 하나의 원칙 수정만), 여전히 검증 및 버전 결정 단계를 수행하세요.

중요한 정보가 없는 경우 (예: 비준 날짜를 정말 모르는 경우), `TODO(<FIELD_NAME>): 설명`을 삽입하고 동기화 영향 보고서의 연기된 항목 아래에 포함하세요.

새 템플릿을 생성하지 마세요; 항상 기존 `.specify/memory/constitution.md` 파일에서 작업하세요.
