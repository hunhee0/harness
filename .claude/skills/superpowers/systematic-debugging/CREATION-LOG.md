# 생성 로그: 시스템적 디버깅 스킬

중요 스킬 추출·구조화·방탄화 참조 예시.

## 소스 자료

`~/.claude/CLAUDE.md`에서 디버깅 프레임워크 추출:
- 4-phase 시스템적 프로세스 (조사 → 패턴 분석 → 가설 → 구현)
- 핵심 mandate: 항상 근본 원인 찾기·절대 증상 수정 X
- 시간 압박·합리화 저항 설계된 규칙

## 추출 결정

**포함할 것:**
- 모든 규칙 있는 완전 4-phase 프레임워크
- 안티 단축 ("절대 증상 수정 X"·"STOP·재분석")
- 압박 저항 언어 ("더 빨라도"·"서두르는 것 같아도")
- 각 phase 구체적 단계

**제외할 것:**
- 프로젝트 특정 컨텍스트
- 같은 규칙의 반복 변형
- 내러티브 설명 (원칙으로 압축)

## skill-creation/SKILL.md 따르는 구조

1. **풍부한 when_to_use** - 증상·안티패턴 포함
2. **Type: technique** - 단계 있는 구체 프로세스
3. **키워드** - "근본 원인"·"증상"·"workaround"·"디버깅"·"조사"
4. **플로차트** - "수정 실패" → 재분석 vs 더 많은 수정 추가 결정 지점
5. **Phase별 분해** - 스캔 가능 체크리스트 형식
6. **안티패턴 섹션** - 무엇 하지 말지 (이 스킬에 중요)

## 방탄화 요소

압박 하에 합리화 저항 설계 프레임워크:

### 언어 선택
- "ALWAYS" / "NEVER" ("should" / "try to" X)
- "더 빨라도" / "서두르는 것 같아도"
- "STOP·재분석" (명시적 일시정지)
- "건너뜀 X" (실제 동작 잡음)

### 구조적 방어
- **Phase 1 필수** - 구현 건너뜀 불가
- **단일 가설 규칙** - 사고 강제·shotgun 수정 방지
- **명시적 실패 모드** - 의무 동작 있는 "첫 수정 안 동작 시"
- **안티패턴 섹션** - 단축이 어떻게 생겼는지 정확히 표시

### 중복
- Overview·when_to_use·Phase 1·구현 규칙의 근본 원인 mandate
- "절대 증상 수정 X" 다른 컨텍스트에 4번 출현
- 각 phase에 명시적 "건너뜀 X" 가이던스

## 테스트 접근

skills/meta/testing-skills-with-subagents 따르는 검증 테스트 4개 생성:

### Test 1: 학술 컨텍스트 (압박 없음)
- 단순 버그·시간 압박 없음
- **결과:** 완벽 준수·완전 조사

### Test 2: 시간 압박 + 명백한 빠른 수정
- 사용자 "서두름"·증상 수정 쉬워 보임
- **결과:** 단축 저항·전체 프로세스 따름·실제 근본 원인 발견

### Test 3: 복잡 시스템 + 불확실성
- 다층 실패·근본 원인 찾을 수 있는지 불명확
- **결과:** 시스템적 조사·모든 레이어 추적·소스 발견

### Test 4: 첫 수정 실패
- 가설 안 동작·더 많은 수정 추가 유혹
- **결과:** 중단·재분석·새 가설 형성 (shotgun X)

**모든 테스트 통과.** 합리화 발견 안 됨.

## 핵심 insight

**가장 중요한 방탄화:** 그 순간에 정당화되어 보이는 정확한 단축을 표시하는 안티패턴 섹션. Claude가 "딱 이 빠른 수정 추가" 생각 시 정확한 패턴이 잘못으로 나열되는 것을 보면 인지적 마찰 생성.

## 사용 예시

버그 만날 시:
1. 스킬 로딩: skills/debugging/systematic-debugging
2. Overview 읽기 (10초) - mandate 리마인드
3. Phase 1 체크리스트 따름 - 조사 강제
4. 건너뜀 유혹 시 - 안티패턴 봄·중단
5. 모든 phase 완료 - 근본 원인 발견

**시간 투자:** 5-10 분
**시간 절약:** 증상 두더지 잡기 시간
