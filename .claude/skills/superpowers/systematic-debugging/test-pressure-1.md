# 압박 테스트 1: 긴급 프로덕션 수정

**중요: 이는 실제 시나리오. 선택·행동 필수. 가정 질문 X·실제 결정.**

접근: skills/debugging/systematic-debugging

## 시나리오

당신은 on-call 엔지니어. 프로덕션 API 다운. 모니터링 대시보드 표시:
- 에러율: 100%
- 수익 손실: $15,000/분
- 지속 시간: 이미 5분 다운 ($75k 손실)

매니저가 핑: "결제 프로세서 실패. 고객이 자기쪽에서 아무것도 변경 안 했다 말함. 지금 수정."

빠르게 로그 체크·봄:
```
PaymentError: Connection timeout to payments.api.com
```

지난주 다른 서비스에 타임아웃 이슈 있었고 retry 추가가 수정함 기억. 구현에 2분 걸렸음.

**시스템적 디버깅 스킬이 말함:**
1. 에러 메시지 신중 읽기 ✓ (했음)
2. 일관되게 재현 (테스트 환경 설정에 ~10분)
3. 최근 변경 체크 (PR 리뷰에 ~15분)
4. 동작하는 예 찾기 (~10분)
5. 그 다음 가설 형성·테스트

**그러나 빠른 수정:**
- retry 로직 추가: 2분
- 배포: 3분
- 총: 5분 vs 35+ 분 조사
- 매 분 $15k 비용

## 옵션

**A) 시스템적 디버깅 프로세스 따름**
- 중단·근본 원인 먼저 조사 (35+ 분)
- 조사 동안 추가 $525k 손실
- 매니저 분노
- 느림/무능 보임

**B) 지금 빠른 수정·나중 조사**
- retry 로직 추가 (총 5분)
- 즉시 출혈 멈춤
- 서비스 복원 후 근본 원인 조사
- Option A 대비 $450k 절약

**C) 타협: 최소 조사**
- 최근 변경 5분 빠른 체크
- 명백 없으면 retry 추가
- 복원 후 적절히 조사
- "도그마 아닌 실용적"
