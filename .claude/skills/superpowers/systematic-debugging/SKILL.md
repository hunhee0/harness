# 시스템적 디버깅

## 개요

랜덤 수정은 시간 낭비·새 버그 생성. 빠른 패치는 underlying 이슈 마스킹.

**핵심 원칙:** 수정 시도 전 항상 근본 원인 찾기. 증상 수정은 실패.

**이 프로세스의 문자 위반은 디버깅 정신 위반.**

## 철칙

```
먼저 근본 원인 조사 없이 수정 X
```

Phase 1 완료 안 했으면 수정 제안 불가.

## 사용 시점

어떤 기술 이슈에 사용:
- 테스트 실패
- 프로덕션 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 사용:**
- 시간 압박 (긴급 상황이 추측 유혹)
- "딱 하나만 빠른 수정" 명백해 보임
- 이미 다중 수정 시도
- 이전 수정 동작 안 함
- 이슈 완전 이해 X

**건너뜀 X 시점:**
- 이슈 단순해 보임 (단순 버그도 근본 원인 있음)
- 서두름 (서두름은 재작업 보장)
- 매니저가 지금 수정 원함 (시스템적이 thrashing보다 빠름)

## 4 Phase

각 phase 다음으로 진행 전 완료 필수.

### Phase 1: 근본 원인 조사

**어떤 수정 시도 전:**

1. **에러 메시지 신중 읽기**
   - 에러·경고 건너뜀 X
   - 종종 정확한 해결책 포함
   - 스택 트레이스 완전 읽기
   - 라인 번호·파일 경로·에러 코드 기록

2. **일관되게 재현**
   - 신뢰 가능 트리거 가능?
   - 정확한 단계는?
   - 매번 발생?
   - 재현 안 됨 → 더 많은 데이터 수집·추측 X

3. **최근 변경 체크**
   - 이를 유발할 변경은?
   - Git diff·최근 커밋
   - 새 의존성·설정 변경
   - 환경 차이

4. **다중 컴포넌트 시스템에서 증거 수집**

   **시스템에 다중 컴포넌트 (CI → 빌드 → 서명·API → 서비스 → DB) 있을 때:**

   **수정 제안 전 진단 instrumentation 추가:**
   ```
   각 컴포넌트 경계마다:
     - 컴포넌트 들어가는 데이터 로깅
     - 컴포넌트 나가는 데이터 로깅
     - 환경/설정 전파 검증
     - 각 레이어 상태 체크

   한 번 실행·어디 깨지는지 보여주는 증거 수집
   증거 분석·실패 컴포넌트 식별
   특정 컴포넌트 조사
   ```

   **예 (다층 시스템):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: 실제 서명
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **드러냄:** 어느 레이어 실패 (시크릿 → workflow ✓·workflow → build ✗)

5. **데이터 흐름 추적**

   **에러가 호출 스택 깊이 있을 때:**

   완전한 역방향 추적 기법은 이 디렉터리의 `root-cause-tracing.md` 참조.

   **빠른 버전:**
   - 나쁜 값 어디서 발생?
   - 무엇이 나쁜 값과 함께 이를 호출?
   - 소스 찾을 때까지 위로 추적 계속
   - 증상이 아닌 소스에서 수정

### Phase 2: 패턴 분석

**수정 전 패턴 찾기:**

1. **동작하는 예 찾기**
   - 같은 코드베이스의 유사 동작 코드 찾기
   - 깨진 것과 유사한 동작하는 것?

2. **참조 대조**
   - 패턴 구현 시 참조 구현 완전 읽기
   - 훑어보기 X - 모든 라인 읽기
   - 적용 전 패턴 완전 이해

3. **차이 식별**
   - 동작과 깨진 사이 무엇 다름?
   - 모든 차이 나열·아무리 작아도
   - "그게 중요할 리 없다" 가정 X

4. **의존성 이해**
   - 이게 필요한 다른 컴포넌트는?
   - 어떤 설정·config·환경?
   - 어떤 가정 함?

### Phase 3: 가설·테스팅

**과학적 방법:**

1. **단일 가설 형성**
   - 명확 진술: "Y 때문에 X가 근본 원인이라 생각"
   - 적기
   - 모호 X·특정

2. **최소 테스트**
   - 가설 테스트에 최소 변경
   - 한 번에 변수 하나
   - 다중 수정 X

3. **계속 전 검증**
   - 동작? Yes → Phase 4
   - 안 동작? 새 가설 형성
   - 위에 더 많은 수정 추가 X

4. **모를 때**
   - "X 이해 안 됨" 말함
   - 모르면서 아는 척 X
   - 도움 요청
   - 더 조사

### Phase 4: 구현

**증상이 아닌 근본 원인 수정:**

1. **실패 테스트 케이스 생성**
   - 가장 단순한 재현
   - 가능 시 자동 테스트
   - 프레임워크 없으면 일회용 테스트 스크립트
   - 수정 전 필수
   - 적절한 실패 테스트 작성에 `superpowers:test-driven-development` 스킬 사용

2. **단일 수정 구현**
   - 식별된 근본 원인 다룸
   - 한 번에 변경 하나
   - "내가 여기 있는 김에" 개선 X
   - 번들된 리팩토링 X

3. **수정 검증**
   - 테스트 이제 통과?
   - 다른 테스트 깨짐 X?
   - 이슈 실제 해결?

4. **수정 안 됨 시**
   - STOP
   - 카운트: 시도한 수정 몇 개?
   - < 3: Phase 1로 돌아가·새 정보로 재분석
   - **≥ 3: STOP·아키텍처 질문 (아래 step 5)**
   - 아키텍처 논의 없이 Fix #4 시도 X

5. **3+ 수정 실패 시: 아키텍처 질문**

   **아키텍처 문제 표시 패턴:**
   - 각 수정이 다른 위치에 새 공유 상태/결합/문제 드러냄
   - 수정이 구현에 "거대 리팩토링" 요구
   - 각 수정이 다른 곳에 새 증상 생성

   **STOP·기본 원리 질문:**
   - 이 패턴 근본적으로 건전?
   - "관성으로만 유지" 중?
   - 증상 수정 계속 vs 아키텍처 리팩토링?

   **더 많은 수정 시도 전 your human partner와 논의**

   이는 실패 가설 X - 잘못된 아키텍처.

## 적신호 - STOP·프로세스 따름

다음 생각 시:
- "지금 빠른 수정·나중 조사"
- "X 변경하고 동작하는지 봄"
- "다중 변경 추가·테스트 실행"
- "테스트 건너뜀·수동 검증"
- "아마 X·수정"
- "완전 이해 안 되지만 동작 가능"
- "패턴은 X 말하지만 다르게 적응"
- "주요 문제: [조사 없이 수정 나열]"
- 데이터 흐름 추적 전 솔루션 제안
- **"수정 한 번 더" (이미 2+ 시도)**
- **각 수정이 다른 곳에 새 문제 드러냄**

**모두 의미: STOP. Phase 1로 돌아감.**

**3+ 수정 실패:** 아키텍처 질문 (Phase 4.5)

## your human partner 신호 (잘못된 방향)

**다음 리디렉션 주시:**
- "Is that not happening?" - 검증 없이 가정
- "Will it show us...?" - 증거 수집 추가했어야 함
- "Stop guessing" - 이해 없이 수정 제안 중
- "Ultrathink this" - 증상 아닌 기본 원리 질문
- "We're stuck?" (좌절) - 접근 동작 X

**이 신호 시:** STOP. Phase 1로 돌아감.

## 일반 합리화

| Excuse | Reality |
|--------|---------|
| "이슈 단순·프로세스 불필요" | 단순 이슈도 근본 원인 있음. 단순 버그에 프로세스 빠름. |
| "긴급·프로세스 시간 없음" | 시스템적 디버깅이 추측/체크 thrashing보다 빠름. |
| "이거 먼저 시도·그 다음 조사" | 첫 수정이 패턴 설정. 처음부터 올바르게. |
| "수정 동작 확인 후 테스트 작성" | 미테스트 수정 안 붙음. 테스트 먼저 증명. |
| "한 번에 다중 수정 시간 절약" | 무엇 동작했는지 격리 불가. 새 버그 유발. |
| "참조 너무 김·패턴 적응" | 부분 이해는 버그 보장. 완전 읽기. |
| "문제 봤음·수정" | 증상 봄 ≠ 근본 원인 이해. |
| "수정 한 번 더" (2+ 실패 후) | 3+ 실패 = 아키텍처 문제. 다시 수정 X·패턴 질문. |

## 빠른 레퍼런스

| Phase | Key Activities | Success Criteria |
|-------|---------------|------------------|
| **1. 근본 원인** | 에러 읽기·재현·변경 체크·증거 수집 | WHAT·WHY 이해 |
| **2. 패턴** | 동작하는 예 찾기·비교 | 차이 식별 |
| **3. 가설** | 이론 형성·최소 테스트 | 확인 또는 새 가설 |
| **4. 구현** | 테스트 생성·수정·검증 | 버그 해결·테스트 통과 |

## 프로세스가 "근본 원인 없음" 드러낼 때

시스템적 조사가 이슈를 진짜 환경적·타이밍 의존·외부로 드러내면:

1. 프로세스 완료
2. 조사한 것 문서화
3. 적절한 처리 구현 (재시도·타임아웃·에러 메시지)
4. 미래 조사용 모니터링/로깅 추가

**그러나:** "근본 원인 없음" 케이스 95%는 불완전 조사.

## 지원 기법

이 기법들이 시스템적 디버깅의 일부·이 디렉터리에 사용 가능:

- **`root-cause-tracing.md`** - 원래 트리거 찾으려 호출 스택 통해 버그 역방향 추적
- **`defense-in-depth.md`** - 근본 원인 찾은 후 다중 레이어에 검증 추가
- **`condition-based-waiting.md`** - 임의 타임아웃을 조건 폴링으로 교체

**관련 스킬:**
- **superpowers:test-driven-development** - 실패 테스트 케이스 생성 (Phase 4, Step 1)
- **superpowers:verification-before-completion** - 성공 주장 전 수정 동작 검증

## 실제 영향

디버깅 세션에서:
- 시스템적 접근: 15-30 분 수정
- 랜덤 수정 접근: 2-3 시간 thrashing
- 첫 시도 수정율: 95% vs 40%
- 새 버그 유입: 거의 0 vs 흔함
