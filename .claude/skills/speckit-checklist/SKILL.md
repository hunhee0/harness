## 체크리스트 목적: "영어를 위한 단위 테스트"

**핵심 개념**: 체크리스트는 **요구사항 작성을 위한 단위 테스트**입니다 — 주어진 도메인에서 요구사항의 품질, 명확성, 완전성을 검증합니다.

**검증/테스트용이 아닌 것**:

- ❌ "버튼 클릭이 올바르게 작동하는지 확인"
- ❌ "오류 처리가 작동하는지 테스트"
- ❌ "API가 200을 반환하는지 확인"
- ❌ 코드/구현이 스펙과 일치하는지 확인

**요구사항 품질 검증용**:

- ✅ "모든 카드 유형에 대한 시각적 계층 요구사항이 정의되어 있는가?" (완전성)
- ✅ "'두드러진 표시'가 구체적인 크기/위치로 정량화되어 있는가?" (명확성)
- ✅ "모든 인터랙티브 요소에 걸쳐 호버 상태 요구사항이 일관성 있는가?" (일관성)
- ✅ "키보드 탐색에 대한 접근성 요구사항이 정의되어 있는가?" (커버리지)
- ✅ "스펙이 로고 이미지 로드 실패 시 동작을 정의하는가?" (엣지 케이스)

**비유**: 스펙이 영어로 작성된 코드라면, 체크리스트는 그것의 단위 테스트 모음입니다. 요구사항이 잘 작성되었는지, 완전한지, 명확하지 않은지, 구현 준비가 되었는지 테스트하는 것이지 — 구현이 작동하는지 테스트하는 것이 아닙니다.

## 사용자 입력

```text
$ARGUMENTS
```

진행 전 반드시 사용자 입력을 고려하세요 (비어있지 않은 경우).

## 실행 단계

1. **설정**: 저장소 루트에서 `.specify/scripts/powershell/check-prerequisites.ps1 -Json`을 실행하고 JSON에서 FEATURE_DIR과 AVAILABLE_DOCS 목록을 파싱합니다.
   - 모든 파일 경로는 절대 경로여야 합니다.
   - args에 'I'm Groot'처럼 작은따옴표가 있는 경우 이스케이프 문법 사용: 예) 'I'\''m Groot' (또는 가능하면 큰따옴표: "I'm Groot").

2. **의도 명확화 (동적)**: spec/plan/tasks에서 추출된 신호를 기반으로 최대 3개의 초기 컨텍스트 명확화 질문 도출 (미리 준비된 카탈로그 없음). 다음을 반드시 지킵니다:
   - 사용자 표현 + spec/plan/tasks에서 추출된 신호로부터 생성
   - 체크리스트 내용을 실질적으로 변경하는 정보에 대해서만 질문
   - `$ARGUMENTS`에서 이미 명확한 경우 개별적으로 건너뜀
   - 폭보다 정밀도 선호

   생성 알고리즘:
   1. 신호 추출: 기능 도메인 키워드 (auth, latency, UX, API 등), 위험 지표 ("critical", "must", "compliance"), 이해관계자 힌트 ("QA", "review", "security team"), 명시적 결과물 ("a11y", "rollback", "contracts").
   2. 신호를 최대 4개의 후보 집중 영역으로 클러스터링 (관련성 순).
   3. 명시적이지 않은 경우 예상 대상 및 타이밍 식별 (작성자, 검토자, QA, 릴리스).
   4. 누락된 차원 탐지: 범위 폭, 깊이/엄격성, 위험 강조, 제외 경계, 측정 가능한 수락 기준.
   5. 다음 원형에서 선택된 질문 작성:
      - 범위 정제 (예: "X와 Y의 통합 접점을 포함해야 하나요, 아니면 로컬 모듈 정확성에 한정하나요?")
      - 위험 우선순위화 (예: "어떤 잠재적 위험 영역이 필수 게이트 검사를 받아야 하나요?")
      - 깊이 보정 (예: "이것이 가벼운 사전 커밋 검사 목록인가요, 공식 릴리스 게이트인가요?")
      - 대상 프레이밍 (예: "작성자만 사용하나요, PR 검토 중 동료들이 사용하나요?")
      - 경계 제외 (예: "이번 라운드에서 성능 튜닝 항목을 명시적으로 제외해야 하나요?")
      - 시나리오 클래스 갭 (예: "복구 흐름이 감지되지 않음 — 롤백/부분 실패 경로가 범위에 포함되나요?")

   질문 포맷 규칙:
   - 옵션을 제시할 때 컬럼이 있는 간결한 테이블 생성: 옵션 | 후보 | 중요성
   - 최대 A–E 옵션; 자유 형식 답변이 더 명확한 경우 테이블 생략
   - 사용자가 이미 말한 것을 다시 말하게 하지 않기
   - 추측적 카테고리 피하기 (환각 없음). 불확실한 경우 명시적으로 묻기: "X가 범위에 포함되는지 확인하세요."

   상호작용이 불가능한 경우 기본값:
   - 깊이: 표준
   - 대상: 코드 관련 시 검토자 (PR); 그 외에는 작성자
   - 집중: 상위 2개 관련성 클러스터

   질문 출력 (Q1/Q2/Q3 레이블). 답변 후: 2개 이상의 시나리오 클래스 (대안/예외/복구/비기능 도메인)가 여전히 불명확하면 한 줄 근거와 함께 최대 2개의 추가 후속 질문 (Q4/Q5) 가능. 총 5개 질문 초과 금지. 사용자가 명시적으로 거부하면 에스컬레이션 건너뜀.

3. **사용자 요청 이해**: `$ARGUMENTS` + 명확화 답변 결합:
   - 체크리스트 테마 도출 (보안, 검토, 배포, ux 등)
   - 사용자가 언급한 명시적 필수 항목 통합
   - 포커스 선택을 카테고리 스캐폴딩에 매핑
   - spec/plan/tasks에서 누락된 컨텍스트 추론 (환각 금지)

4. **기능 컨텍스트 로드**: FEATURE_DIR에서 읽기:
   - spec.md: 기능 요구사항 및 범위
   - plan.md (있는 경우): 기술 세부사항, 의존성
   - tasks.md (있는 경우): 구현 태스크

   **컨텍스트 로드 전략**:
   - 활성 집중 영역과 관련된 필요한 부분만 로드 (전체 파일 덤프 피하기)
   - 긴 섹션을 간결한 시나리오/요구사항 불릿으로 요약 선호
   - 점진적 공개 사용: 갭 감지 시에만 후속 검색 추가
   - 소스 문서가 큰 경우 원본 텍스트 임베딩 대신 중간 요약 항목 생성

5. **체크리스트 생성** - "요구사항 단위 테스트" 생성:
   - 없는 경우 `FEATURE_DIR/checklists/` 디렉토리 생성
   - 고유한 체크리스트 파일명 생성:
     - 도메인 기반의 짧고 설명적인 이름 사용 (예: `ux.md`, `api.md`, `security.md`)
     - 형식: `[domain].md`
   - 파일 처리 동작:
     - 파일이 **없는** 경우: 새 파일 생성하고 CHK001부터 번호 부여
     - 파일이 **있는** 경우: 기존 파일에 새 항목 추가, 마지막 CHK ID에서 이어서 시작 (예: 마지막이 CHK015면 CHK016부터 시작)
   - 기존 체크리스트 내용 절대 삭제/대체하지 않음 - 항상 보존하고 추가

   **핵심 원칙 - 구현이 아닌 요구사항 테스트**:
   모든 체크리스트 항목은 **요구사항 자체**를 다음 측면에서 평가해야 합니다:
   - **완전성**: 필요한 모든 요구사항이 있는가?
   - **명확성**: 요구사항이 명확하고 구체적인가?
   - **일관성**: 요구사항이 서로 정렬되어 있는가?
   - **측정 가능성**: 요구사항을 객관적으로 검증할 수 있는가?
   - **커버리지**: 모든 시나리오/엣지 케이스가 다루어졌는가?

   **카테고리 구조** - 요구사항 품질 차원으로 그룹화:
   - **요구사항 완전성** (필요한 모든 요구사항이 문서화되어 있는가?)
   - **요구사항 명확성** (요구사항이 구체적이고 명확한가?)
   - **요구사항 일관성** (요구사항이 충돌 없이 정렬되어 있는가?)
   - **수락 기준 품질** (성공 기준이 측정 가능한가?)
   - **시나리오 커버리지** (모든 흐름/케이스가 다루어졌는가?)
   - **엣지 케이스 커버리지** (경계 조건이 정의되어 있는가?)
   - **비기능 요구사항** (성능, 보안, 접근성 등 - 명세되어 있는가?)
   - **의존성 및 가정** (문서화되고 검증되어 있는가?)
   - **모호성 및 충돌** (무엇이 명확화가 필요한가?)

   **체크리스트 항목 작성 방법 - "영어 단위 테스트"**:

   ❌ **틀림** (구현 테스트):
   - "랜딩 페이지에 3개의 에피소드 카드가 표시되는지 확인"
   - "데스크탑에서 호버 상태가 작동하는지 테스트"
   - "로고 클릭이 홈으로 이동하는지 확인"

   ✅ **맞음** (요구사항 품질 테스트):
   - "추천 에피소드의 정확한 수와 레이아웃이 명세되어 있는가?" [완전성]
   - "'두드러진 표시'가 구체적인 크기/위치로 정량화되어 있는가?" [명확성]
   - "모든 인터랙티브 요소에 걸쳐 호버 상태 요구사항이 일관성 있는가?" [일관성]
   - "모든 인터랙티브 UI에 대한 키보드 탐색 요구사항이 정의되어 있는가?" [커버리지]
   - "로고 이미지 로드 실패 시 폴백 동작이 명세되어 있는가?" [엣지 케이스]
   - "비동기 에피소드 데이터의 로딩 상태가 정의되어 있는가?" [완전성]
   - "경쟁하는 UI 요소에 대한 시각적 계층이 정의되어 있는가?" [명확성]

   **항목 구조**:
   각 항목은 다음 패턴을 따라야 합니다:
   - 요구사항 품질에 대해 묻는 질문 형식
   - 스펙/플랜에 **작성된** (또는 작성되지 않은) 내용에 집중
   - 대괄호에 품질 차원 포함 [완전성/명확성/일관성 등]
   - 기존 요구사항 확인 시 스펙 섹션 참조 `[Spec §X.Y]`
   - 누락된 요구사항 확인 시 `[Gap]` 마커 사용

   **품질 차원별 예시**:

   완전성:
   - "모든 API 실패 모드에 대한 오류 처리 요구사항이 정의되어 있는가? [Gap]"
   - "모든 인터랙티브 요소에 대한 접근성 요구사항이 명세되어 있는가? [완전성]"
   - "반응형 레이아웃에 대한 모바일 브레이크포인트 요구사항이 정의되어 있는가? [Gap]"

   명확성:
   - "'빠른 로딩'이 구체적인 타이밍 임계값으로 정량화되어 있는가? [명확성, Spec §NFR-2]"
   - "'관련 에피소드' 선택 기준이 명시적으로 정의되어 있는가? [명확성, Spec §FR-5]"
   - "'두드러진'이 측정 가능한 시각적 속성으로 정의되어 있는가? [모호성, Spec §FR-4]"

   일관성:
   - "모든 페이지에 걸쳐 탐색 요구사항이 정렬되어 있는가? [일관성, Spec §FR-10]"
   - "랜딩 페이지와 상세 페이지 간에 카드 컴포넌트 요구사항이 일관성 있는가? [일관성]"

   커버리지:
   - "제로 상태 시나리오(에피소드 없음)에 대한 요구사항이 정의되어 있는가? [커버리지, 엣지 케이스]"
   - "동시 사용자 인터랙션 시나리오가 다루어졌는가? [커버리지, Gap]"
   - "부분 데이터 로딩 실패에 대한 요구사항이 명세되어 있는가? [커버리지, 예외 흐름]"

   측정 가능성:
   - "시각적 계층 요구사항이 측정/테스트 가능한가? [수락 기준, Spec §FR-1]"
   - "'균형 잡힌 시각적 무게'를 객관적으로 검증할 수 있는가? [측정 가능성, Spec §FR-2]"

   **시나리오 분류 및 커버리지** (요구사항 품질 집중):
   - 다음에 대한 요구사항이 있는지 확인: 주요, 대안, 예외/오류, 복구, 비기능 시나리오
   - 각 시나리오 클래스에 대해: "[시나리오 유형] 요구사항이 완전하고 명확하고 일관성 있는가?"
   - 시나리오 클래스 누락 시: "[시나리오 유형] 요구사항이 의도적으로 제외되었나요, 아니면 누락되었나요? [Gap]"
   - 상태 변이가 발생할 때 복원력/롤백 포함: "마이그레이션 실패에 대한 롤백 요구사항이 정의되어 있는가? [Gap]"

   **추적 가능성 요구사항**:
   - 최소: 항목의 ≥80%에 추적 가능성 참조 포함 필수
   - 각 항목은 다음을 참조해야 함: 스펙 섹션 `[Spec §X.Y]`, 또는 마커 사용: `[Gap]`, `[모호성]`, `[충돌]`, `[가정]`
   - ID 체계가 없는 경우: "요구사항 및 수락 기준 ID 체계가 확립되어 있는가? [추적 가능성]"

   **이슈 표면화 및 해결** (요구사항 품질 문제):
   요구사항 자체에 대해 묻기:
   - 모호성: "'빠른'이 구체적인 지표로 정량화되어 있는가? [모호성, Spec §NFR-1]"
   - 충돌: "§FR-10과 §FR-10a 간에 탐색 요구사항이 충돌하는가? [충돌]"
   - 가정: "'항상 사용 가능한 팟캐스트 API'의 가정이 검증되었는가? [가정]"
   - 의존성: "외부 팟캐스트 API 요구사항이 문서화되어 있는가? [의존성, Gap]"
   - 누락된 정의: "'시각적 계층'이 측정 가능한 기준으로 정의되어 있는가? [Gap]"

   **내용 통합**:
   - 소프트 상한: 원본 후보 항목이 40개를 초과하면 위험/영향 순으로 우선순위화
   - 같은 요구사항 측면을 확인하는 유사 중복 항목 병합
   - 낮은 영향 엣지 케이스가 5개를 초과하면 하나의 항목 생성: "요구사항에서 엣지 케이스 X, Y, Z가 다루어졌는가? [커버리지]"

   **🚫 절대 금지** - 이것들은 구현 테스트로 만듦:
   - ❌ "확인", "테스트", "검증" + 구현 동작으로 시작하는 항목
   - ❌ 코드 실행, 사용자 액션, 시스템 동작 참조
   - ❌ "올바르게 표시됨", "제대로 작동함", "기대대로 기능함"
   - ❌ "클릭", "탐색", "렌더링", "로드", "실행"
   - ❌ 테스트 케이스, 테스트 계획, QA 절차
   - ❌ 구현 세부사항 (프레임워크, API, 알고리즘)

   **✅ 필수 패턴** - 이것들이 요구사항 품질 테스트:
   - ✅ "[요구사항 유형]이 [시나리오]에 대해 정의/명세/문서화되어 있는가?"
   - ✅ "[모호한 용어]가 구체적인 기준으로 정량화/명확화되어 있는가?"
   - ✅ "[섹션 A]와 [섹션 B] 간에 요구사항이 일관성 있는가?"
   - ✅ "[요구사항]을 객관적으로 측정/검증할 수 있는가?"
   - ✅ "요구사항에서 [엣지 케이스/시나리오]가 다루어졌는가?"
   - ✅ "스펙이 [누락된 측면]을 정의하는가?"

6. **구조 참조**: `.specify/templates/checklist-template.md`의 정식 템플릿을 따라 체크리스트 생성 (제목, 메타 섹션, 카테고리 제목, ID 형식). 템플릿이 없는 경우 사용: H1 제목, purpose/created 메타 줄, CHK001부터 시작하는 전역 증가 ID가 있는 `- [ ] CHK### <요구사항 항목>` 줄이 포함된 `##` 카테고리 섹션.

7. **보고**: 체크리스트 파일 전체 경로, 항목 수 출력, 실행이 새 파일을 생성했는지 기존 파일에 추가했는지 요약. 요약:
   - 선택된 집중 영역
   - 깊이 레벨
   - 행위자/타이밍
   - 통합된 사용자 지정 필수 항목

**중요**: 각 `/speckit-checklist` 명령 실행은 짧고 설명적인 체크리스트 파일명을 사용하며 새 파일을 생성하거나 기존 파일에 추가합니다. 이를 통해:

- 다양한 유형의 여러 체크리스트 (예: `ux.md`, `test.md`, `security.md`)
- 체크리스트 목적을 나타내는 단순하고 기억하기 쉬운 파일명
- `checklists/` 폴더에서 쉬운 식별 및 탐색

불필요한 항목을 피하려면 설명적인 유형을 사용하고 완료 후 오래된 체크리스트를 정리하세요.

## 안티 예시: 하지 말아야 할 것

**❌ 틀림 - 이것들은 요구사항이 아닌 구현을 테스트:**

```markdown
- [ ] CHK001 - 랜딩 페이지에 3개의 에피소드 카드가 표시되는지 확인 [Spec §FR-001]
- [ ] CHK002 - 데스크탑에서 호버 상태가 올바르게 작동하는지 테스트 [Spec §FR-003]
- [ ] CHK003 - 로고 클릭이 홈 페이지로 이동하는지 확인 [Spec §FR-010]
- [ ] CHK004 - 관련 에피소드 섹션이 3-5개 항목을 표시하는지 확인 [Spec §FR-005]
```

**✅ 맞음 - 이것들이 요구사항 품질 테스트:**

```markdown
- [ ] CHK001 - 추천 에피소드의 수와 레이아웃이 명시적으로 명세되어 있는가? [완전성, Spec §FR-001]
- [ ] CHK002 - 모든 인터랙티브 요소에 걸쳐 호버 상태 요구사항이 일관성 있게 정의되어 있는가? [일관성, Spec §FR-003]
- [ ] CHK003 - 모든 클릭 가능한 브랜드 요소에 대한 탐색 요구사항이 명확한가? [명확성, Spec §FR-010]
- [ ] CHK004 - 관련 에피소드 선택 기준이 문서화되어 있는가? [Gap, Spec §FR-005]
- [ ] CHK005 - 비동기 에피소드 데이터에 대한 로딩 상태 요구사항이 정의되어 있는가? [Gap]
- [ ] CHK006 - "시각적 계층" 요구사항을 객관적으로 측정할 수 있는가? [측정 가능성, Spec §FR-001]
```

**핵심 차이점:**

- 틀림: 시스템이 올바르게 작동하는지 테스트
- 맞음: 요구사항이 올바르게 작성되어 있는지 테스트
- 틀림: 동작 검증
- 맞음: 요구사항 품질 검증
- 틀림: "X를 하는가?"
- 맞음: "X가 명확하게 명세되어 있는가?"
