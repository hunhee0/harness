## 1. 개요 / 목적

이메일·비밀번호로 사용자를 인증하고 세션 토큰(JWT)을 발급한다. 기존 사용자 테이블 재사용, 신규 가입은 범위 외.

## 2. 사용자 스토리

- 등록된 사용자로서, 이메일·비밀번호를 제출하면 액세스 토큰을 받아 보호된 API를 호출할 수 있다.
- 잘못된 자격 증명 제출 시, 어떤 필드가 틀렸는지 노출하지 않는 일반 오류를 받는다(계정 열거 방지).

## 3. 기능 요구사항 (FR)

| ID | 요구사항 |
|----|---------|
| FR-1 | `POST /auth/login` — `{email, password}` 수신, 성공 시 `{access_token, token_type}` 반환 |
| FR-2 | 비밀번호는 해시 비교(bcrypt 등). 평문 저장·비교 금지 |
| FR-3 | 실패 시 401 + 일반 메시지("invalid credentials"). 필드별 구분 금지 |
| FR-4 | 토큰 만료 시간 설정(기본 30분). 만료 경계는 `<=` 가 아닌 `<` 로 비교 |
| FR-5 | 로그인 시도 rate limit (예: 분당 5회/IP) |

## 4. 비기능 요구사항 (NFR)

- 응답 p95 < 300ms (해시 비용 포함)
- 시크릿(JWT 서명 키)은 환경 변수/시크릿 매니저. 하드코딩 금지.

## 5. 성공 기준 (검증 가능)

- [ ] 유효 자격 증명 → 200 + 유효 JWT (디코딩 시 sub=user_id, exp 존재)
- [ ] 무효 자격 증명 → 401 + 일반 메시지
- [ ] 만료 토큰 → 보호 엔드포인트에서 401
- [ ] rate limit 초과 → 429
- [ ] 단위·통합 테스트 커버리지 ≥ 80%

## 6. 범위 밖 (Out of Scope)

- 회원 가입, 비밀번호 재설정, 소셜 로그인, 리프레시 토큰 회전.

## 7. 미해결 / NEEDS CLARIFICATION

- (예) 리프레시 토큰 필요 여부 — 현재 범위 밖으로 가정.
