## 에러 유형별 복구 경로

### 1. 빌드/타입 오류

```
빌드 실패
  → 에러 메시지 분석 (Gather Context)
  → 수정 시도 (Take Action)
  → 빌드 재실행 (Verify)
  → 2회 실패: /clear 후 명확한 프롬프트 재작성
```

### 2. 테스트 실패

```
테스트 실패
  → 실패 테스트 격리 확인 (다른 테스트에 영향?)
  → 구현 로직 재검토 (spec.md 대비)
  → 수정 → 재실행
  → 2회 반복 실패: implementer 에이전트 재호출 + 실패 컨텍스트 전달
```

**중요**: 통과를 위해 테스트를 수정하지 말 것. 스펙이 틀리지 않은 한 구현을 수정한다.

### 3. 에이전트 루프 / 반복 실패

```
에이전트 교정 2회 실패
  → /clear
  → 오케스트레이터에 에스컬레이션
  → 실패한 태스크를 더 작은 단위로 분해
  → 재시도
```

### 4. 컨텍스트 과부하

증상: 이전 지시 무시, 일관성 없는 응답.

```
  → /compact 시도 (컨텍스트 압축)
  → 효과 없으면 /clear
  → tasks.md 현재 상태 먼저 읽기
  → 필요한 규칙/스펙만 명시적으로 참조하여 재시작
```

### 5. 구현 중 스펙 불일치 발견

구현하다 스펙이 잘못된 것을 발견한 경우:

```
  → 구현 즉시 중단
  → question 툴로 사용자 보고: 현상 + 옵션 (스펙 수정 / 구현 방향 변경)
  → 사용자 결정 후 spec.md 먼저 업데이트
  → 이후 구현 재개
```

절대: 스펙을 무시하고 임의로 구현하지 않는다.

## 롤백 절차

### 코드 롤백

```bash
# 이력 보존 롤백 (권장)
git revert {commit-hash}

# 주의: 다음은 위험 — 사용자 명시 동의 필요
git reset --hard {commit}
git push --force
```

### 스펙 롤백

스펙을 수정했다가 되돌려야 할 경우:
1. `docs/spec/{feature}/spec.md` 이전 버전 복원 (git 이용)
2. 영향받은 `plan.md`, `tasks.md` 함께 검토
3. changelog에 롤백 사유 기록

## Human-in-the-loop 강제 조건

다음은 자동 진행 금지 — 반드시 `question` 툴로 사용자 확인:

- 파일/디렉토리 삭제
- DB 스키마 변경 (migration)
- `git reset --hard` / `git push --force`
- 스펙 major 변경 (범위 확대/축소)
- 외부 서비스 연동 (API 키, 환경 설정 변경)
